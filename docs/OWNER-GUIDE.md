# Руководство владельца — ulkan

Простыми словами: где что лежит, что делать при проблемах, как поручать задачи ИИ без программиста. Технические подробности — в `docs/TZ.md` и `docs/METHOD-ulkan.md`, но здесь они не нужны.

---

## 1. Карта хозяйства (что где живёт и как туда зайти)

| Что | Где | Зачем / что там смотреть |
|---|---|---|
| **Код** | GitHub → репозиторий `Ulan136/ulkan`, ветка **`main`** | вся программа; изменения делает ИИ и «пушит» сюда |
| **Сайт (прод)** | `ulkan.vercel.app` | рабочий сайт для сотрудников и клиентов |
| **Панель хостинга** | `vercel.com` → проект **ulkan** | тут видно деплои (выкладки), логи, переменные |
| **База данных** | `console.neon.tech` → проект/ветка ulkan | все данные: заказы, пользователи, склад |
| **Real-time** | `dashboard.pusher.com` → приложение **ulkan** | «живые» обновления экранов; вкладка **Debug Console** — видно события |

**Как это связано:** ИИ меняет код → «пушит» в GitHub → **Vercel автоматически выкладывает** новую версию на `ulkan.vercel.app` (примерно **2 минуты**). База (Neon) и Pusher — отдельные сервисы, к ним сайт подключается по «переменным окружения».

### ⚠ Про базу — очень важно

**В этой же базе Neon живёт ВТОРОЙ проект — касса.** Поэтому менять структуру базы можно **только вручную** через SQL в Neon. Команда `prisma db push` (автоматическое изменение структуры) **ЗАПРЕЩЕНА** — она может испортить таблицы кассы. Если ИИ предлагает `db push` — отказать (см. §3).

### Переменные окружения (ключи и пароли доступа)

Vercel → проект ulkan → **Settings → Environment Variables**. Там лежат:
- `DATABASE_URL` — строка подключения к базе Neon;
- `PUSHER_*` и `NEXT_PUBLIC_PUSHER_*` — ключи real-time;
- `AUTH_SECRET` — секрет для входа (сессии).

**После любого изменения переменной обязателен Redeploy** (переменные «вшиваются» в сборку): Vercel → Deployments → у последней выкладки «…» → **Redeploy**.

## 2. Что делать если… (по реальным случаям)

### 😴 Сайт пишет «нет связи с базой» / всё висит
База Neon «уснула» (бесплатный тариф засыпает без нагрузки).
1. Откройте `console.neon.tech`, войдите, выберите проект ulkan.
2. Слева **SQL Editor** → впишите `SELECT 1;` → **Run**.
3. Подождите минуту, обновите сайт (Ctrl+Shift+R). База проснётся.

### 🫥 Портал пустой / кнопки пропали
1. **Жёсткое обновление**: Ctrl+Shift+R (Windows) / Cmd+Shift+R (Mac).
2. Если вверху «**Сессия устарела**» — нажмите «Выйти и войти заново».
3. Проверьте выкладку: Vercel → **Deployments** → верхняя должна быть **Ready** и на нужном коммите (последнее изменение).

### 🚧 Изменения «не выходят» на сайт
Vercel мог застрять на старой версии.
1. Vercel → **Deployments**.
2. Найдите верхнюю со статусом **Ready** — посмотрите, какой это коммит (совпадает с последним в GitHub?).
3. Если завис/ошибка — у нужной выкладки «…» → **Redeploy**; либо у прошлой хорошей → **Promote to Production**.

### 🔙 Нужно откатить плохую версию
1. Vercel → **Deployments**.
2. Найдите **предыдущую** выкладку со статусом **Ready** (которая работала).
3. «…» → **Promote to Production**. Сайт мгновенно вернётся к ней.

### 🔎 Данные выглядят неправильно — SQL-шпаргалка
Neon → **SQL Editor**, вставьте нужный запрос, **Run** (замените значения в кавычках):

Посмотреть карточку с позициями:
```sql
SELECT * FROM "Order" WHERE id = 'C-730-010726';
SELECT id, name1c, oral, qty, price, status, resp, supplier, leg
FROM "Position" WHERE "cardId" = 'C-730-010726' ORDER BY "createdAt";
```
Сколько номенклатуры в справочнике:
```sql
SELECT COUNT(*) FROM "Nomenclature";
```
Позиции конкретного логиста «в работе»:
```sql
SELECT p.id, p.name1c, p.status, p.leg, o.id AS card
FROM "Position" p JOIN "Order" o ON o.id = p."cardId"
WHERE p.resp = 'Имя Логиста' AND p.status <> 'Доставлено';
```
Черновик смены логиста за сегодня:
```sql
SELECT * FROM "DailyReport" WHERE status = 'draft' ORDER BY date DESC LIMIT 10;
```

## 3. Как ставить задачи ИИ (Claude Code) без программиста

### Золотые правила (коротко)
1. **Одна задача за раз** — не смешивать несколько дел в одном запросе.
2. **Каждый запрос заканчивать**: «npx tsc --noEmit → npm run build → commit → push».
3. **Изменения структуры базы — только вручную**: попросить ИИ **вывести SQL**, выполнить его самому в Neon. Никакого `db push`.
4. **После push — проверить сайт руками** (Ctrl+Shift+R на нужном экране).
5. Требовать **отчёт**: причина проблемы, что изменено, как проверить.

### Шаблон постановки задачи (копировать и заполнить)
```
Задача: <что нужно, простыми словами>.
Контекст/пример: <номер карточки, экран, роль — где видно проблему>.
Требования:
  - <конкретное поведение 1>
  - <конкретное поведение 2>
  - не трогать: <что не должно измениться, напр. порталы логиста/клиента>
Если нужна правка структуры базы — выведи SQL, я выполню сам в Neon (db push НЕ делать).
В конце: npx tsc --noEmit → npm run build → отчёт (причина + что изменено + как проверить) →
commit "<короткое описание>" → push.
```

### 🛑 Стоп-сигналы — если ИИ предлагает такое, ОСТАНОВИТЬ и разобраться
- `prisma db push`, `prisma migrate`, `--accept-data-loss` — **опасно для кассы**, отказать.
- Массовое удаление (`DELETE FROM …` без узкого условия, «удалить всех …») — уточнить, зачем.
- Коммит файла `.env` / `.env.txt` — там секреты, **нельзя** в репозиторий.
- «Сбросить/пересоздать базу», «drop table» — только с явным пониманием последствий.

Правильная реакция: «Стоп. Объясни простыми словами, что именно и зачем, какие данные затронет, есть ли безопасная альтернатива».

## 4. Доступы-памятка (без секретов — только ГДЕ)

Пароли/ключи здесь **не пишем** — только где лежат и чем логинимся.

| Сервис | Вход | Где ключи/секреты |
|---|---|---|
| **GitHub** (код) | аккаунт `Ulan136` | — |
| **Vercel** (хостинг) | обычно вход через GitHub | ключи проекта — Vercel → Settings → Environment Variables |
| **Neon** (база) | аккаунт Neon (email или GitHub/Google) | строка `DATABASE_URL` — в Neon (Connection) и в Vercel Env |
| **Pusher** (real-time) | аккаунт Pusher | App Keys — в Pusher → App Keys и в Vercel Env |

**При утере доступа:** восстановить через «Забыли пароль» на сайте сервиса (почта аккаунта). Ключи Pusher/Neon можно перевыпустить в их панелях — но тогда обновить их в Vercel Env и сделать **Redeploy**.

## 5. Незакрытые хвосты (что и зачем)

### 🔴 Обязательно перед реальными клиентами
- **Сбросить пароли в Neon** — раньше список пользователей вместе с зашифрованными паролями мог быть виден лишним людям (уязвимость закрыта в коде, но старые пароли лучше сменить).
- **Удалить файл `.env.txt`** — в нём секреты; он не должен попасть в общий доступ.
- **Разнести кассу и ulkan по разным базам** — сейчас они в одной базе Neon, из-за чего опасно менять структуру. Пока живут вместе — строго только ручной SQL.

### 🟡 Улучшения (не срочно)
- Экран «Номенклатура» не обновляется «вживую» при правках — нужно обновить страницу вручную.
- «Детали проекта» по клику — кнопку убрали, саму страницу-карточку проекта ещё не сделали.
- Суб-пользователи (несколько логинов на одну компанию) — в базе поддержаны, но формы в интерфейсе нет.
- Мелкие технические долги перечислены в `docs/periphery-audit.md`.
