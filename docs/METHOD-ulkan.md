Как безопасно и предсказуемо вести разработку именно этого проекта. Читать вместе с `docs/TZ.md` (что за система) и `docs/OWNER-GUIDE.md` (для владельца-непрограммиста).

## 1. Архитектура: слои и «куда класть новый код»

```
UI (components/*.tsx)  ──вызов──▶  API route (app/api/**/route.ts)
                                        │  тонкий: гард → парсинг → делегирование
                                        ▼
                          services/*   +   lib/*        ──▶  prisma  ──▶  Postgres (Neon)
                     (бизнес-логика)     (утилиты, чистые)
```

**Правила размещения:**

| Что | Куда | Пример |
|---|---|---|
| Новый переход/действие карточки | `services/orderWorkflow.ts` → `TRANSITIONS[action]` | новый возврат, новый статус-переход |
| Складская операция | `services/stockOps.ts` | резерв/списание/приход |
| Определение плеча по поставщику | `services/legDetection.ts` | — |
| Статусные предикаты позиции | `lib/positionState.ts` (клиент-безопасно) | isHandedOff, myActivePos |
| Метрики (прогресс/суммы) | `lib/orderMetrics.ts` | cardProgress, cardSum, PCT |
| Строковые статусы/экраны | `lib/orderStatus.ts` | POS_STATUS, CARD_STATUS, SCREENS |
| Гард/сессия | `lib/auth.ts` | requireSession |
| Real-time сигнал (сервер) | `lib/pusherServer.ts` `pushSignal` | — |
| Real-time подписка (клиент) | `lib/live.ts` `useLiveData` | — |
| Даты | `lib/dates.ts` (локальные формы) / `lib/reportDay.ts` (день Алматы) | — |
| Клиентский вызов API | `lib/api.ts` (обёртка `req()` с обработкой ошибок) | — |
| Роут API | `app/api/**/route.ts` — **только**: гард → распарсить → вызвать сервис → `pushSignal` → ответ | — |

**Принцип**: роут тонкий, логика — в `services`/`lib`. Не дублировать формулы прогресса/сумм и статус-литералы — брать из `orderMetrics`/`orderStatus`.

## 2. Рабочий процесс

**Одна задача = один коммит.** Каждый промпт заканчивается циклом:

```
npx tsc --noEmit   →   npm run build   →   smoke (проверить логику)   →   commit   →   push   →   проверка прода
```

- **tsc и build обязаны быть зелёными** до коммита. `strict:false` — часть ошибок типов не ловится, поэтому логику проверяем глазами/скриптом.
- **Коммиты**: осмысленное сообщение (что и почему), формат как в истории (`feat:`/`fix:`/`refactor:`/`chore:`/`security:`/`docs:`). Ко-автор: `Co-Authored-By: Claude Opus 4.8 (1M context)`.
- **git add точечный**: перечислять файлы явно, **никогда не коммитить `.env.txt`** (см. §5).
- **Проверка прода**: push → Vercel деплоит ~2 мин → открыть `ulkan.vercel.app`, проверить нужный экран (жёсткое обновление Ctrl+Shift+R).

### ⛔ Изменения схемы БД — ТОЛЬКО ручной SQL в Neon

**В той же базе живёт ВТОРОЙ проект (касса).** Поэтому:

- **НИКОГДА** `prisma db push`, `prisma migrate`, `--accept-data-loss` — они могут удалить/переписать чужие таблицы.
- Изменение схемы: сначала правим `schema.prisma` (для типов Prisma Client), затем **вручную выполняем эквивалентный SQL** в Neon SQL Editor.
- ИИ **обязан вывести готовый SQL в отчёте** (`ALTER TABLE …`), владелец выполняет его сам в Neon.
- После ручного SQL — `npx prisma generate` (только генерация клиента, без обращения к БД).

## 3. Грабли проекта (из реальной истории)

| Грабля | Симптом | Правильно |
|---|---|---|
| **Serverless + fire-and-forget** | сигнал не долетает, обновление с задержкой 40–60с | `await pushSignal(...)` **после всех записей БД, перед `return`** (`8bb33f6`, `dca4e00`). На Vercel функция «замерзает» после ответа. |
| **UTC vs Алматы** | отчёт/дата «съезжает» на день | день = `almatyDay()` (`lib/reportDay.ts`), не `new Date('YYYY-MM-DD')` (UTC-полночь). Дефолты форм — `todayLocal()`. |
| **editingRef-паузы** | ввод в форме сбрасывается при live-обновлении | `useLiveData(..., pausedRef)`; ref снимать при закрытии редактора/смене вкладки/размонтировании (иначе залипнет → обновления встанут). |
| **select без password** | утечка хэшей паролей | user-запросы всегда с `select` без `password`; списки пользователей — только админам (`064c5b4`). |
| **Страховочный поллинг** | если WS Pusher пропустил ивент — «висит» | `PollingTransport` 20с как страховка + visibilitychange/focus/online (`lib/live.ts`). Код работает **и без Pusher-ключей**. |
| **Битые id/имя** | пусто/не находит | `UnifiedSelect` отдаёт **имя**, не id — не искать `find(u=>u.id===name)`. Сравнение имён — `eqName` (trim+lowercase). |
| **leg vs статусные предикаты** | у старых данных `leg` битый → неверная вкладка/кнопка | UI филиала — по **статусу** (`lib/positionState.ts`), не по `leg`. Сервер `leg` только **проставляет** (для фильтра логиста). |
| **Vercel env = новый билд** | поменяли переменную — не применилось | после изменения Environment Variables нужен **Redeploy** (env вшивается в билд). |
| **`.env` vs `.env.txt`** | риск коммита секретов / не тот DATABASE_URL | локальный `.env.txt` **не коммитить**; он мог указывать на ЧУЖУЮ базу — никаких `db push`. |
| **Пустые порталы** | branch/logist видят пусто | правильные роль-гарды (`['branch']`/`['logist']`) + `include.positions.where` по имени + JWT-fallback роли (`391ce9f`, `5c23d03`). |
| **Компонент внутри компонента = мина** | вложенный чат/редактор теряет фокус, сбрасывает стейт, «залипает» на загрузке | `PosCard`/`OrderCard` были объявлены **внутри** порталов → на каждый рендер родителя React видит новый тип функции и **размонтирует** поддерево карточки (петля ремонтирования `CardChat`: `load()` → `onCount` → новый объект `setMsgCount` → рендер → ремонт → снова `load()`; `1a3f146`). **Правило**: карточные/тяжёлые/stateful-компоненты объявлять на **верхнем уровне модуля**, состояние передавать пропсами. |
| **Тихая пустота v3** | вечная «Загрузка…» вместо данных/пусто/ошибки (третье пришествие грабли) | у любого async-состояния **три исхода на экране**: данные / пусто / ошибка. `finally { setLoaded(true) }`, пустой ответ → «Сообщений пока нет» с активным вводом, не-ok → текст ошибки. «Загрузка…» держится **ровно на время запроса** (`1a3f146`, ср. `reports/daily` «Нет отчётов» вместо 500). |

Полный список находок периферии — `docs/periphery-audit.md`.

## 4. Чек-лист новой фичи

Перед коммитом ответить «да» по каждому применимому пункту:

- [ ] **Гард**: роут закрыт `requireSession(req, roles?)`? Мутация справочника/пользователей — только `['super_admin','bookkeeper']`? Данные пользователей — `select` без `password`?
- [ ] **pushSignal**: мутирую данные → шлю `await pushSignal(канал)` после всех записей? Канал верный (`orders`/`reports`/`settings`)? Подписчик реально перегружает эти данные?
- [ ] **History**: изменение карточки/позиции журналируется в `History`?
- [ ] **Паузы ввода**: если live-обновление может перерисовать открытую форму — `pausedRef` подключён и не залипает?
- [ ] **Async-состояние = три исхода**: у каждого загружаемого блока на экране есть **данные / пусто / ошибка** (нет «вечной загрузки»)?
- [ ] **Стабильность компонента**: карточные/stateful-компоненты объявлены на верхнем уровне модуля (не внутри родителя → не ремонтируются на каждый рендер)?
- [ ] **Статусы/метрики**: беру из `orderStatus`/`orderMetrics`/`positionState`, не дублирую литералы и формулы?
- [ ] **Даты**: день Алматы через `almatyDay`, дефолты через `todayLocal`, не `toISOString().slice(0,10)`?
- [ ] **Схема БД**: если менял — вывел SQL в отчёт, `db push` не трогал?
- [ ] **Прод-проверка**: после push проверил на `ulkan.vercel.app` реальным сценарием?
- [ ] **Отчёт**: причина, что изменено, как проверить.

## 5. Карта хвостов

### 🔴 Перед живыми клиентами (критично)

- **Reset password в Neon** — до коммита `064c5b4` `users`/`settings` GET отдавали bcrypt-хэши любому авторизованному; хэши могли утечь. Сбросить пароли пользователей.
- **Удалить `.env.txt`** — локальный файл с секретами/строкой подключения; не должен попасть в репозиторий/утечь.
- **Касса — в отдельную БД** — сейчас проект кассы живёт в той же базе Neon, из-за чего запрещён `prisma db push`. Разнести по разным БД, чтобы миграции ulkan не угрожали кассе.
